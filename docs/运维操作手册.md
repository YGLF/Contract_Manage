# 合同管理系统微服务运维操作手册

## 1. 手册用途

本手册用于系统已经部署完成后的日常运维。

适合处理这些工作：

1. 查看服务状态
2. 重启单个服务
3. 查看日志
4. 处理常见故障
5. 做简单巡检

## 2. 运维前先确认你在哪个目录

进入项目目录：

```bash
cd E:/HTGL/AnXin_Contract_Manage_microservices_v1.1
```

后续所有 `docker compose` 命令都建议在这个目录执行。

如果是新手，优先记住这 3 个脚本：

- `bash scripts/check-env.sh`
- `bash scripts/start-microservices.sh`
- `bash scripts/health-check.sh`

如果运维对象是生产镜像路径，请改用：

- `COMPOSE_FILE=./docker-compose.production.yml bash scripts/check-env.sh`
- `docker compose --env-file .env -f docker-compose.production.yml ps`
- `COMPOSE_FILE=./docker-compose.production.yml bash scripts/health-check.sh`

CentOS 7 仅作为应用系统部署测试环境，不作为实际生产基线。

## 3. 最常用的 6 个命令

### 3.1 查看所有服务状态

```bash
docker compose -f docker-compose.microservices.yml ps
```

生产镜像路径：

```bash
docker compose -f docker-compose.production.yml ps
```

什么时候用：

- 想看哪些服务正常启动
- 想看哪些服务退出了

### 3.2 启动全部服务

```bash
bash scripts/start-microservices.sh
```

生产镜像路径：

```bash
docker compose --env-file .env -f docker-compose.production.yml up -d
```

### 3.3 停止全部服务

```bash
docker compose -f docker-compose.microservices.yml down
```

生产镜像路径：

```bash
docker compose -f docker-compose.production.yml down
```

### 3.4 重启单个服务

```bash
docker compose -f docker-compose.microservices.yml restart gateway-service
docker compose -f docker-compose.microservices.yml restart contract-service
```

生产镜像路径：

```bash
docker compose -f docker-compose.production.yml restart gateway-service
docker compose -f docker-compose.production.yml restart contract-service
```

### 3.5 查看全部日志

```bash
docker compose -f docker-compose.microservices.yml logs --tail=200
```

如果只是想先确认系统还活着，优先执行：

```bash
bash scripts/health-check.sh
```

生产镜像路径：

```bash
COMPOSE_FILE=./docker-compose.production.yml bash scripts/health-check.sh
```

### 3.6 查看单个服务日志

```bash
docker compose -f docker-compose.microservices.yml logs gateway-service
docker compose -f docker-compose.microservices.yml logs contract-service
docker compose -f docker-compose.microservices.yml logs report-service
```

## 4. 服务状态怎么看

执行：

```bash
docker compose -f docker-compose.microservices.yml ps
```

常见状态解释：

- `Up`
  - 服务正常运行
- `Exited`
  - 服务已经退出，通常说明启动失败或异常终止
- `Restarting`
  - 服务反复重启，通常说明依赖配置有问题

如果看到 `Exited` 或 `Restarting`：

1. 先看这个服务日志
2. 不要先重启全部服务
3. 先定位是哪一个配置或依赖出问题

## 5. 日志排查怎么做

### 5.1 先看最后 200 行

```bash
docker compose -f docker-compose.microservices.yml logs --tail=200 report-service
```

### 5.2 如果日志刷得太快

实时跟踪：

```bash
docker compose -f docker-compose.microservices.yml logs -f gateway-service
```

### 5.3 新手看日志时重点找这些词

- `connection refused`
- `access denied`
- `permission denied`
- `panic`
- `failed`
- `forbidden`
- `unauthorized`
- `not configured`

## 6. 健康检查怎么做

### 6.1 网关健康检查

```bash
bash scripts/health-check.sh
```

这个脚本会自动检查：

- 网关 `/health`
- 网关路由表
- 未登录访问业务接口是否正确返回 `401`
- 是否存在异常状态容器
- `unhealthy` 容器也会被视为失败

### 6.2 路由检查

```bash
curl http://127.0.0.1:8100/gateway/routes
```

### 6.3 鉴权检查

```bash
curl http://127.0.0.1:8100/api/contracts/contracts
```

预期：

- 返回 `401`
- 表示网关鉴权正常

## 7. 常见运维场景

### 7.1 某个服务起不来

处理顺序：

1. 执行：

```bash
docker compose -f docker-compose.microservices.yml ps
```

2. 找到异常服务

3. 查看它的日志：

```bash
docker compose -f docker-compose.microservices.yml logs <service-name>
```

4. 根据日志判断：

- 数据库问题
- 依赖 URL 配错
- 权限或目录问题
- 端口冲突

### 7.2 修改了 `.env` 之后怎么生效

执行：

```bash
docker compose -f docker-compose.microservices.yml down
docker compose --env-file .env -f docker-compose.microservices.yml up -d
```

生产镜像路径：

```bash
docker compose -f docker-compose.production.yml down
docker compose --env-file .env -f docker-compose.production.yml up -d
```

如果只是单个服务受影响，也可以只重启该服务，但新手建议整体重启一遍。

### 7.3 上传文件失败

先看：

1. `document-service` 日志
2. `.env` 中 `UPLOAD_DIR`
3. 容器挂载目录是否存在

可检查：

```bash
docker compose -f docker-compose.microservices.yml logs document-service
```

### 7.4 导出失败

先看：

1. `report-service` 日志
2. `approval-workflow-service` 日志
3. 当前审批单是否已审批
4. 审批单是否过期或已消费

### 7.5 AI 助手不可用

先看：

1. `search-ai-service` 日志
2. `AI_MODEL_ENDPOINT` 是否配置正确
3. 910B 接口是否可达

测试：

```bash
curl http://910b-inference.internal:9000/infer
```

## 8. 常见报错处理表

### 8.1 `connection refused`

含义：

- 目标服务没启动
- 目标地址对了，但端口没人监听

怎么处理：

1. 检查目标服务是否 `Up`
2. 检查 `*_SERVICE_URL`
3. 检查数据库或 910B 是否启动

### 8.2 `access denied`

含义：

- 账号密码错误
- 数据库账号无权限

怎么处理：

1. 重新核对 `.env`
2. 请数据库管理员确认授权

### 8.3 `permission denied`

含义：

- 上传目录无权限
- 挂载目录无权限
- 容器无写入权限

怎么处理：

1. 检查目录是否存在
2. 检查目录权限
3. 检查挂载路径是否正确
4. 生产镜像路径下确认 `UPLOAD_DIR=uploads` 并映射到 `/app/uploads`

### 8.4 `service target is not configured`

含义：

- 某个依赖 URL 没配

怎么处理：

1. 检查 `.env`
2. 检查 compose 环境变量注入

### 8.5 `401 unauthorized`

含义：

- 没登录
- token 错
- 密钥不一致

怎么处理：

1. 检查是否带 token
2. 检查 `JWT_SECRET`
3. 检查身份服务和网关配置是否一致

### 8.6 `403 forbidden`

含义：

- 权限不足
- 部门数据范围不匹配
- 导出审批单无效

怎么处理：

1. 检查用户权限
2. 检查数据范围
3. 检查审批状态

### 8.7 `502 bad gateway`

含义：

- 网关访问下游失败

怎么处理：

1. 查看下游服务是否正常
2. 检查网关里的服务地址
3. 查看对应服务日志

## 9. 日常巡检建议

建议每天至少做一次：

1. 查看服务状态

```bash
docker compose -f docker-compose.microservices.yml ps
```

2. 查看关键服务日志是否有大量错误

```bash
docker compose -f docker-compose.microservices.yml logs --tail=100 gateway-service
docker compose -f docker-compose.microservices.yml logs --tail=100 contract-service
docker compose -f docker-compose.microservices.yml logs --tail=100 report-service
```

3. 检查网关健康

```bash
curl http://127.0.0.1:8100/health
```

4. 检查上传目录是否正常

5. 检查数据库是否可连接

## 10. 如果完全不会排查，按这个顺序来

遇到问题时，不要先重装，不要先改代码，按这个顺序处理：

1. 看 `.env` 是否填错
2. 看 `docker compose ps`
3. 看 `docker compose logs <服务名>`
4. 看数据库和 910B 地址是否通
5. 再决定是否需要重新启动服务

只要按这个顺序处理，大多数问题都能先定位出来。
