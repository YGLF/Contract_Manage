# 合同管理系统微服务部署手册

## 1. 手册用途

本手册面向第一次接触本项目的实施人员、运维人员和开发人员。

目标是让一个新手按步骤完成以下工作：

1. 准备部署环境
2. 配置微服务环境变量
3. 启动全部微服务
4. 检查服务是否正常
5. 在遇到报错时知道先看哪里、怎么处理

## 2. 本系统部署前提

当前交付环境前提如下：

- 操作系统：麒麟 V10 SP3
- 数据库：南大金仓 GBADE，按 MySQL 兼容模式接入
- AI 推理：910B 算力服务器，提供 HTTP 接口
- 部署目录：[AnXin_Contract_Manage_microservices_v1.1](E:/HTGL/AnXin_Contract_Manage_microservices_v1.1)

系统边界说明：

- 只管理已盖章生效合同
- 不做合同起草、签署、电子签、用印
- 审批服务只承接允许存在的审批子流程

## 3. 部署前你需要准备什么

### 3.1 你至少需要知道这几项信息

请先向现场环境负责人确认：

1. GBADE 数据库地址、端口、用户名、密码、数据库名
2. 910B 推理接口地址
3. 当前服务器是否已安装 Docker 和 Docker Compose
4. 当前服务器是否可以访问数据库和 910B 服务
5. 服务器是否允许开放 `8100` 端口给前端或 Nginx 访问

如果这几项信息不明确，不建议直接开始部署。

### 3.2 机器最低检查

在服务器终端执行以下命令：

```bash
uname -a
docker --version
docker compose version
```

预期结果：

- 能看到麒麟系统信息
- `docker --version` 正常返回版本
- `docker compose version` 正常返回版本

如果报错：

- `docker: command not found`
  - 说明 Docker 未安装，需要先安装 Docker
- `docker compose: not a docker command`
  - 说明 Docker Compose 未安装，或者版本太旧

### 3.3 目录检查

进入项目目录：

```bash
cd E:/HTGL/AnXin_Contract_Manage_microservices_v1.1
```

如果部署到 Linux 标准目录，可以先复制到例如：

```bash
/opt/contract-manage-microservices
```

确认目录中存在以下关键内容：

- `cmd/`
- `internal/`
- `pkg/`
- `frontend/`
- `docs/`
- `scripts/`
- `docker-compose.microservices.yml`
- `.env.microservices.example`

建议优先使用这 3 个脚本：

- `scripts/check-env.sh`：启动前检查环境和配置
- `scripts/start-microservices.sh`：一键启动全部微服务
- `scripts/health-check.sh`：启动后做基础健康检查

## 4. 第一步：配置环境变量

### 4.1 复制环境模板

在项目根目录执行：

```bash
cp .env.microservices.example .env
```

如果是 Windows PowerShell：

```powershell
Copy-Item .env.microservices.example .env
```

### 4.2 修改 `.env`

请重点修改以下几项：

```env
JWT_SECRET=replace-with-strong-secret
ADMIN_USERNAME=admin
ADMIN_PASSWORD=replace-with-strong-admin-password
AUDIT_ADMIN_USERNAME=auditadmin
AUDIT_ADMIN_PASSWORD=replace-with-strong-audit-password

DB_HOST=10.0.0.10
DB_PORT=3306
DB_USER=contract_user
DB_PASSWORD=replace-password
DB_NAME=contract_manage

AI_MODEL_ENDPOINT=http://910b-inference.internal:9000/infer
UPLOAD_DIR=uploads
```

必须修改说明：

1. `JWT_SECRET`
   - 不能用默认值
   - 建议换成一串长随机字符串
2. `ADMIN_PASSWORD`、`AUDIT_ADMIN_PASSWORD`
   - 必须显式设置，系统不再提供默认管理员密码
   - 必须使用现场密钥管理或密码策略生成的强口令
3. `DB_HOST`、`DB_PORT`、`DB_USER`、`DB_PASSWORD`、`DB_NAME`
   - 必须改成现场 GBADE 信息
4. `AI_MODEL_ENDPOINT`
   - 必须改成 910B 实际推理接口地址
5. `UPLOAD_DIR`
   - 建议保持为 `uploads`
   - 开发/联调 compose 会映射到联调容器内的默认上传目录
   - 生产镜像 compose 会映射到 `/app/uploads`

### 4.3 新手最容易犯的错误

错误 1：数据库填了本机 `localhost`

- 如果数据库不在当前机器上，不能写 `localhost`
- 必须写真实数据库 IP 或域名

错误 2：端口写错

- GBADE 如果不是 `3306`，必须按现场实际值填写

错误 3：AI 地址少写了 `http://`

- `AI_MODEL_ENDPOINT` 必须是完整 URL

## 5. 第二步：检查数据库是否可连通

### 5.1 用最简单方式检查网络连通

先检查端口通不通：

```bash
ping 10.0.0.10
telnet 10.0.0.10 3306
```

如果服务器没有 `telnet`，可尝试：

```bash
nc -zv 10.0.0.10 3306
```

预期结果：

- `ping` 能通
- `3306` 端口能连上

如果失败：

- `ping` 不通
  - 先确认网络、路由、防火墙
- 端口不通
  - 先确认数据库服务是否启动
  - 再确认数据库安全组/防火墙是否放通

### 5.2 检查数据库账号是否可用

生产部署前，数据库和应用账号应由 DBA 或现场密钥管理流程预先创建。项目根目录的 `init.sql` 只负责创建业务数据库，不再携带默认数据库密码，也不执行 `GRANT ALL PRIVILEGES` 这类过宽授权。请按最小权限原则为 `DB_USER` 授权，仅允许访问 `DB_NAME` 对应库内本系统需要的对象。

如果现场有数据库客户端，可执行：

```bash
mysql -h 10.0.0.10 -P 3306 -u contract_user -p
```

输入密码后如果能进入数据库，再执行：

```sql
USE contract_manage;
SHOW TABLES;
```

如果报错：

- `Access denied`
  - 用户名或密码错误
  - 数据库账号权限不足
- `Unknown database`
  - `DB_NAME` 填错了
  - 或数据库还没创建

### 5.3 推荐先执行自动检查脚本

在项目根目录执行：

```bash
bash scripts/check-env.sh
```

这个脚本会检查：

- Docker 和 Docker Compose 是否可用
- `.env` 是否存在
- 关键环境变量是否还是默认占位值
- 数据库端口是否可连接
- 910B 推理地址是否可访问

如果输出中有 `FAIL`，先修复后再继续。

## 6. 第三步：检查 910B 推理接口

执行：

```bash
curl http://910b-inference.internal:9000/infer
```

如果返回不是标准业务结果没关系，关键是：

- 能连通
- 不是 DNS 错误
- 不是连接超时

如果报错：

- `Could not resolve host`
  - 域名解析失败，改用实际 IP，或修复 DNS
- `Connection refused`
  - 910B 推理服务未启动
- `Connection timed out`
  - 网络或防火墙未放通

## 7. 第四步：启动微服务

### 7.1 启动命令

推荐直接执行：

```bash
bash scripts/start-microservices.sh
```

这个脚本会先检查环境，再执行启动，并等待网关健康检查通过。

如果你要手工启动，再执行下面这条命令：

```bash
docker compose --env-file .env -f docker-compose.microservices.yml up -d
```

生产部署优先使用预编译镜像路径：

```bash
COMPOSE_FILE=./docker-compose.production.yml bash scripts/check-env.sh
docker compose --env-file .env -f docker-compose.production.yml build
docker compose --env-file .env -f docker-compose.production.yml up -d
COMPOSE_FILE=./docker-compose.production.yml bash scripts/health-check.sh
```

`docker-compose.production.yml` 使用 `Dockerfile.microservice` 为每个微服务构建独立镜像，不挂载源码，也不在容器启动时执行 `go run`。`docker-compose.microservices.yml` 仍保留给现场联调和问题复现使用。
如果希望复用现有检查脚本验证生产 compose，请在执行 `check-env.sh` 和 `health-check.sh` 时显式传入 `COMPOSE_FILE=./docker-compose.production.yml`。

当前生产 compose 已为关键 HTTP 微服务统一配置容器级 `/health` 探针，包括 `gateway`、`identity`、`audit`、`contract`、`document`、`approval-workflow`、`notification`、`report`、`search-ai`。这些探针用于判断进程是否存活，不等价于数据库、下游微服务、910B 模型接口都已就绪。

### 7.2 启动后查看状态

执行：

```bash
docker compose -f docker-compose.microservices.yml ps
```

预期结果：

- 各服务状态为 `Up`
- `gateway-service` 应逐步进入 `healthy`

一期重点关注这些服务：

- `gateway-service`
- `identity-service`
- `audit-service`
- `contract-service`
- `document-service`
- `approval-workflow-service`
- `report-service`
- `search-ai-service`

### 7.3 如果启动失败怎么办

执行：

```bash
docker compose -f docker-compose.microservices.yml logs --tail=200
```

或查看单个服务：

```bash
docker compose -f docker-compose.microservices.yml logs gateway-service
docker compose -f docker-compose.microservices.yml logs contract-service
docker compose -f docker-compose.microservices.yml logs report-service
```

常见失败场景：

#### 场景 1：端口被占用

错误表现：

- `bind: address already in use`

处理方法：

1. 查占用进程：

```bash
netstat -tlnp | grep 8100
```

2. 停掉占用程序，或修改 `.env` 中对应端口

#### 场景 2：数据库连接失败

错误表现：

- `connection refused`
- `access denied`
- `unknown database`

处理方法：

1. 重新检查 `.env`
2. 检查 GBADE 是否启动
3. 检查数据库账号权限
4. 检查服务器到数据库的网络

#### 场景 3：某个服务依赖地址未配置

错误表现：

- `service target is not configured`
- `failed dependency`

处理方法：

1. 检查 `.env` 中相关 `*_SERVICE_URL`
2. 检查 `docker-compose.microservices.yml` 是否正确引用了这些变量

#### 场景 4：上传目录不可写

错误表现：

- `failed to prepare upload directory`
- `permission denied`

处理方法：

1. 确认 `UPLOAD_DIR` 路径存在
2. 确认容器内路径可写
3. 必要时手工创建目录并赋权

## 8. 第五步：健康检查

### 8.1 检查网关健康

推荐直接执行：

```bash
bash scripts/health-check.sh
```

这个脚本会自动检查：

- 网关 `/health`
- 网关路由表
- 未登录访问业务接口是否正确返回 `401`
- 是否存在 `Exited` 或 `Restarting` 容器

当前 `docker-compose.microservices.yml` 已为所有微服务补充 `restart: unless-stopped`，并为 `gateway-service` 配置容器级 `/health` 探针。它的作用是帮助现场更快识别网关未就绪或异常退出；但它不能替代业务验收，也不能替代数据库、910B、审批链路这些外部依赖检查。

`docker-compose.production.yml` 对关键 HTTP 微服务也统一复用了同一套 `/health` 容器探针，适合做基础 liveness 判断；`outbox-dispatcher` 作为操作型作业不参与常驻健康探针。如果后续需要更严格的 readiness，应单独补充数据库、下游依赖和外部模型可用性检查。

如果希望复用现有检查脚本验证生产 compose，请在执行 `check-env.sh` 和 `health-check.sh` 时显式传入 `COMPOSE_FILE=./docker-compose.production.yml`。现场如果只有 `docker-compose`，脚本会自动回退，但麒麟生产环境仍建议统一使用 `docker compose`。

如果你要手工逐项检查，再执行下面这些命令。

执行：

```bash
curl http://127.0.0.1:8100/health
```

预期结果：

返回 `status=ok` 类似内容。

### 8.2 检查服务路由表

执行：

```bash
curl http://127.0.0.1:8100/gateway/routes
```

预期结果：

应能看到各微服务的路由地址映射。

### 8.3 检查未登录访问拦截

执行：

```bash
curl http://127.0.0.1:8100/api/contracts/contracts
```

预期结果：

- 返回 `401`
- 说明网关鉴权生效

如果返回不是 `401`：

- 检查 `gateway-service` 是否正确加载 `JWT_SECRET`
- 检查路由是否走了 `protected` 分组

## 9. 第六步：部署后最低验收

请至少做以下检查：

1. 所有容器状态正常
2. `gateway-service` 健康检查正常
3. 未带 token 访问业务接口返回 `401`
4. 上传目录可用
5. 审计服务正常
6. 910B 地址可访问
7. 导出审批链路可用

## 10. 常见报错与处理方法

### 10.1 `docker compose up -d` 后服务反复重启

原因通常是：

- 数据库连不上
- 环境变量缺失
- 端口冲突

处理方法：

```bash
docker compose -f docker-compose.microservices.yml ps
docker compose -f docker-compose.microservices.yml logs <service-name>
```

### 10.2 `401 unauthorized`

原因：

- 未带 token
- token 无效
- `JWT_SECRET` 不一致

处理方法：

1. 检查是否正确登录拿到 token
2. 检查网关和身份服务是否使用同一 `JWT_SECRET`

### 10.3 `403 forbidden`

原因：

- 权限不足
- 部门数据范围不匹配
- 导出审批单无效、已过期或已消费

处理方法：

1. 检查当前账号权限
2. 检查 `X-User-Permissions` 或 token 中权限声明
3. 检查审批单状态是否为 `approved`

### 10.4 `502 bad gateway`

原因：

- 网关下游服务未启动
- 服务之间 URL 配错
- 某个服务内部报错

处理方法：

1. 检查 `docker compose ps`
2. 检查下游服务日志
3. 检查 `.env` 中所有 `*_SERVICE_URL`

### 10.5 `failed to validate export approval`

原因：

- `approval-workflow-service` 未启动
- `APPROVAL_WORKFLOW_SERVICE_URL` 配错
- 审批单不存在

处理方法：

1. 检查审批服务状态
2. 检查 `REPORT_SERVICE_URL` 和 `APPROVAL_WORKFLOW_SERVICE_URL`
3. 重新申请导出审批单

## 11. 如果你完全不会排查，最少看这三处

出问题时，先看下面三项，不要先改代码：

1. `.env` 是否填写正确
2. `docker compose ps` 是否有服务没起来
3. `docker compose logs <服务名>` 的最后 100 行

多数部署失败问题，都能从这三处先定位出来。
